## Executive Verdict

**Stack:** Next.js 14 (App Router) + React 18 + TypeScript 5 + Firebase 12 (Auth / Firestore / Storage) + Tailwind + React Query + Zustand.
**Hosting target:** Vercel (inferred — no Vercel config yet).
**Git:** `git@github.com:MorganDevInSA/ironmind.git`, single `main` branch.
**Maturity:** **Solid app code + Firebase rules, but zero delivery pipeline.** The product layer is well-architected (three-layer data hardening, typed services, rules + indexes committed). What is missing is the _platform_ around it: no MCP tooling, no CI, no typecheck/test scripts, no preview/prod separation, no Vercel config, no Firebase project context, no secret rotation story, no monitoring.

**Top 5 risks (ordered):**

1. **`.firebaserc` points to placeholder `YOUR_PROJECT_ID_HERE`** — any `firebase deploy` will fail or (worse) target the wrong project.
2. **No CI** — nothing blocks broken builds, type errors, or lint failures from reaching `main`.
3. **No Vercel CLI + no `vercel.ts`/`vercel.json`** — deploys, previews, envs, and rollbacks are all manual/unreproducible.
4. **No MCP tooling configured** (`.cursor/mcp.json` missing) — agent can't inspect Vercel/Firebase or fetch up-to-date docs.
5. **No `typecheck` / `test` / `format` scripts** in `package.json` — no safety net for refactors.

**Quick wins (< 30 min total):** 1.1, 1.2, 1.3, 2.1, 2.2, 5.1.

### [ ] 4.3 — Deploy Firestore rules & indexes from CI

**Why:** Rules/indexes drift silently in the dashboard. Make CI the source of truth.
**File:** `.github/workflows/firebase-rules.yml`

```yaml
name: Deploy Firebase Rules & Indexes
on:
  push:
    branches: [main]
    paths:
      - 'firestore.rules'
      - 'firestore.indexes.json'
      - 'storage.rules'
      - 'firebase.json'
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm i -g firebase-tools
      - run: firebase deploy --only firestore:rules,firestore:indexes,storage:rules --project ironmindmp --non-interactive
        env:
          GOOGLE_APPLICATION_CREDENTIALS: ${{ github.workspace }}/sa.json
        # Generate a Service Account with "Firebase Rules Admin" + "Cloud Datastore Index Admin"
        # Store the JSON in GH secret FIREBASE_SERVICE_ACCOUNT, then write it before the deploy step:
      - run: echo '${{ secrets.FIREBASE_SERVICE_ACCOUNT }}' > sa.json
```

> Swap the last two steps' order so `sa.json` is written **before** the deploy. Shown above for clarity.

# The Publish Command (NEW — Session 3)

**What you asked for:** A single terminal command to push the latest dev state live so users can see it.

**The command:**

```bash
npm run publish
```

**What it does (end-to-end):**

1. Checks git status — aborts if you have uncommitted changes (commit or stash first)
2. Shows current branch + HEAD commit
3. Runs `npm run ci` (lint → typecheck → build) with live output — aborts if any stage fails
4. Pushes current branch to GitHub (`git push origin HEAD`)
5. Prints the appropriate URL:
   - **If on `main`**: production URL (`https://ironmind-morgans-projects-bc4d5795.vercel.app`) — Vercel deploys in ~60s
   - **If on any other branch**: preview dashboard URL — Vercel builds a preview deployment visible in the dashboard

**Vercel auto-deploy behavior:**

- Vercel's GitHub integration (already linked) watches the repo
- Push to `main` → production deploy
- Push to any other branch → preview deploy
- No manual `vercel deploy` needed — the GitHub push triggers it

**Rollback if needed:**

```bash
vercel promote <previous-deployment-url>    # instant, zero rebuild
```

**Implementation:** `scripts/publish.mjs` — Node ESM, no external deps, uses `child_process` to shell out.
