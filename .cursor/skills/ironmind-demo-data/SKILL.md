# IRONMIND demo data (personas + synthetic history)

**Canonical reference:** [`Documentation/EXPERT-DEMO-DATA-AND-STORAGE-GUIDE.md`](../../../Documentation/EXPERT-DEMO-DATA-AND-STORAGE-GUIDE.md)

## Hard rules

1. **Collection paths:** only `collections.*` from [`src/lib/firebase/config.ts`](../../../src/lib/firebase/config.ts).
2. **Writes:** synthetic history must use **services** (`createWorkout`, `saveNutritionDay`, `saveRecoveryEntry`, …) as in [`src/lib/seed/demo-historical.ts`](../../../src/lib/seed/demo-historical.ts) — never raw Firestore from pages or one-off scripts.
3. **Orchestration:** any new `src/lib/seed/*.ts` module must be **imported and invoked** from [`src/lib/seed/index.ts`](../../../src/lib/seed/index.ts); demo UI flows use `seedMortonData` / `seedCheriData` / `seedAlexData` / `seedJordanData` / `seedFezData` / `seedMariaData` (see [`src/components/onboarding/DemoProfileModal.tsx`](../../../src/components/onboarding/DemoProfileModal.tsx)).
4. **Types:** shapes live in [`src/lib/types/index.ts`](../../../src/lib/types/index.ts); import JSON packs must stay aligned with [`src/services/import.service.ts`](../../../src/services/import.service.ts).
5. **Check-ins (demo overwrite):** Weekly scale/tape **story** for the six roster personas is **hand-authored** in [`src/lib/seed/demo-data/physique/`](../../../src/lib/seed/demo-data/physique/) (`DEMO_PHYSIQUE_WEEKLY_BY_PERSONA`, **`getDemoPhysiqueWeeks`**). **`seedDemoHistoricalData`** is the **only** consumer for production-shaped demos; it calls **`deleteAllCheckIns`** first, then writes each week via `saveCheckIn`. **Coach notes** are twelve **distinct** strings per persona (not copy-paste). Intermediate weeks get **deterministic** micro-jitter on literals for realistic UI noise; the **final** check-in uses **raw** row values so profile parity holds. Edit the persona `*.ts` literals when changing the underlying arc.
6. **Physique writes:** [`physique.service.ts`](../../../src/services/physique.service.ts) `saveCheckIn` merges new `measurements` onto existing docs for that date and runs **`sanitizeMeasurementsInput`** (`src/lib/utils/measurement-bounds.ts`) on incoming fields. Dashboard / physique charts use **`measurementForChart`** and **chronological sort + `dateKey` (`YYYY-MM-DD`)** for series keys so ticks and tooltips stay unambiguous.
7. **Physique History (app):** The `/physique` page shows a **table** of recent check-ins (newest first; **first 10** rows from the loaded list). **Δ** values are computed **client-side** vs the **next older** row for scale kg and each tape site both rows logged — not a separate stored field. Demo **deterministic jitter** on intermediate weeks is intentional so the table does not read like a perfect spreadsheet; keep weekly literals aligned with the coaching arc.

## Realism checklist

- Imperfect adherence, macro drift, sleep/HRV waves, optional **deload** or **stress** weeks (e.g. Cheri week 3, Maria week 5 in generator).
- Avoid impossible body composition velocity; keep **one active program / phase** per user (use existing training/coaching services when creating programs and phases).
- Do not claim events in **journal** or **coachNotes** that the seed loop does not approximate (e.g. “full miss week” without lowered session frequency).

## Re-seed

Changing TypeScript does not mutate production data. Re-run the **demo profile modal** (or a controlled dev call to `seed*Data`) against **emulator or disposable users** — see `README_DATA_LAYER.md` and the expert guide §10.
